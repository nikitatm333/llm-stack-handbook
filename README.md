## 1. LLM

**LLM (Large Language Model, большая языковая модель)** — авторегрессионная нейронная сеть, обученная на корпусе текстовых данных с целью моделирования условного распределения вероятностей следующего токена в последовательности. Формально модель параметризует вероятность:

```
P(tₙ | t₁, t₂, ..., tₙ₋₁)
```

где `t₁...tₙ₋₁` — предшествующие токены контекста, `tₙ` — предсказываемый токен.

**Токен** — минимальная единица обработки текста в LLM. Токен не эквивалентен слову: токеном может быть часть слова, знак препинания, пробел или целое слово в зависимости от алгоритма токенизации. Например, слово «tokenization» разбивается как `["token", "ization"]`. Средняя длина токена составляет ~4 символа в английском тексте и ~6 символов в русском. Скорость генерации измеряется в **tok/s** (токенов в секунду).

> пример курса токенов к словам https://platform.openai.com/tokenizer

Модель не «понимает» текст в семантическом смысле — она статистически предсказывает наиболее вероятное продолжение входной последовательности на основе распределений, усвоенных при обучении.

## 2. Архитектура локального LLM INFERENCE


```
┌─────────────────────────────────────────┐
│           Пользователи и доступ         │
└──────────────────┬──────────────────────┘
                   │ 
┌──────────────────▼──────────────────────┐
│                  UI                     │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│              API слой                   │
└──────────────────┬──────────────────────┘
                   │
┌──────────────────▼──────────────────────┐
│           Движок модели                 │
└──────────────────┬──────────────────────┘
                   │ 
┌──────────────────▼──────────────────────┐
│            Модель LLM                   │
└─────────────────────────────────────────┘
```

**Уровень модели** — файл весов в формате GGUF. Хранит параметры нейронной сети, метаданные архитектуры и словарь токенизатора.

**Уровень движка** — inference runtime, загружает веса в память, выполняет forward pass, управляет KV-кэшем, предоставляет API.

**API слой** — OpenAI-совместимый REST API. Любой клиент, поддерживающий OpenAI API, может использовать локальную модель без изменений в коде.

**UI** — веб-интерфейс для конечного пользователя. Реализует управление диалогами, аутентификацию, RAG, историю запросов.

Разделение уровней позволяет заменять компоненты независимо: сменить модель без изменения движка, сменить UI без изменения API, переключиться с локального inference на облачный сервис изменением одного параметра конфигурации.

## 3. Модель LLM

### 3.1 Параметры модели (веса)

**Нейронная сеть** — математическая функция, реализованная как последовательность слоёв линейных преобразований и нелинейных функций активации. Каждый слой принимает на вход вектор чисел и возвращает другой вектор.

> Активации — это промежуточные данные, которые нейросеть вычисляет в процессе обработки запроса.

**Линейное преобразование** — умножение входного вектора на матрицу весов с добавлением вектора смещений:

```
y = W · x + b

x — входной вектор (числовое представление токена)
W — матрица весов (параметры, усвоенные при обучении)
b — вектор смещений (bias)
y — выходной вектор
```

**Тензор** — многомерный массив чисел. Скаляр — тензор нулевого ранга, вектор — первого, матрица — второго. В нейронных сетях данные представлены тензорами: например, батч из 8 последовательностей по 2048 токенов, каждый из которых представлен вектором из 4096 чисел — это тензор размерности [8, 2048, 4096].

**Параметр (вес)** — отдельное числовое значение типа с плавающей точкой внутри матрицы W или вектора b. Совокупность всех параметров определяет поведение модели и является результатом обучения методом градиентного спуска.

**Количество параметров** — основная характеристика размера модели:

```
1.5B параметров  =  1 500 000 000 чисел
7B параметров    =  7 000 000 000 чисел
32B параметров   = 32 000 000 000 чисел
```

При полной точности (FP32, 4 байта на параметр) модель с 7B параметров занимает **28 GB** памяти. Для уменьшения требований к памяти применяется квантизация.

---

### 3.2 Квантизация

**Квантизация** — процесс снижения числовой точности представления весов модели с целью уменьшения объёма занимаемой памяти и ускорения вычислений.

**Форматы точности:**

| Формат | Байт на параметр | Биты | Примечание |
|---|---|---|---|
| FP32 (float32) | 4 | 32 | Полная точность, эталон |
| FP16 (float16) | 2 | 16 | Стандарт  |
| BF16 (bfloat16) | 2 | 16 | Тот же диапазон что FP32, меньшая точность мантиссы |
| INT8 | 1 | 8 | Агрессивная квантизация |
| Q4_K_M | ~0.5 | ~4.5 | Нелинейная, рекомендуемый баланс |
| Q2_K | ~0.31 | ~2.6 | Нелинейная, заметная деградация |

**Механизм Q4-квантизации:**

Веса группируются блоками (обычно 32 значения). Для каждого блока вычисляется масштабный коэффициент (scale) и минимальное значение. Каждый вес представляется 4-битным целым числом (0-15). При вычислениях восстанавливается приближённое значение:

```
вес ≈ int4_value × scale + min
```

**Влияние квантизации:**

*Объём VRAM* — снижается пропорционально битности:
```
7B FP16   = ~14 GB
7B Q4_K_M = ~4.5 GB
7B Q2_K   = ~3.0 GB
```

*Качество ответов* — существенно снижается при агрессивной квантизации:
- Q8/Q6 — практически идентично FP16, потери незначительны
- Q4_K_M — рекомендуемый баланс качества и объёма
- Q2_K — заметная деградация, повышенная частота галлюцинаций

*Скорость inference* — при Q4 скорость может быть выше FP16 за счёт уменьшения объёма данных, читаемых из памяти за единицу времени (memory-bound операция).

**Суффиксы в названиях GGUF файлов:**
- `Q4_K_M` — 4-битная квантизация, метод K-quants, Medium вариант
- `Q2_K` — 2-битная квантизация, метод K-quants
- `Q4_0` — 4-битная квантизация, базовый метод (устаревший, хуже Q4_K_M)
- `_K_S`, `_K_M`, `_K_L` — Small/Medium/Large варианты метода

---

### 3.3 Форматы хранения моделей

**GGUF (GPT-Generated Unified Format)** — бинарный формат хранения весов модели, разработанный командой llama.cpp. Единый файл содержит:

- Все веса модели в выбранном формате точности
- Метаданные архитектуры: количество слоёв, размер скрытого состояния, тип attention механизма
- Словарь токенизатора
- Гиперпараметры: контекстное окно, тип нормализации, параметры RoPE

Самодостаточность формата позволяет запускать модель без внешних зависимостей и конфигурационных файлов.

---

### 3.4 KV-кэш

**Механизм Self-Attention** — ключевой компонент архитектуры Transformer. При генерации каждого нового токена модель вычисляет его зависимость от всех предшествующих токенов контекста. Для каждого токена вычисляются три вектора:

- **Query (Q)** — «запрос» текущего токена
- **Key (K)** — «ключ» каждого токена в контексте
- **Value (V)** — «значение» каждого токена в контексте

Attention score вычисляется как:

```
Attention(Q, K, V) = softmax(QKᵀ / √dₖ) · V
```

**KV-кэш (Key-Value Cache)** — буфер в VRAM, хранящий уже вычисленные векторы K и V для всех токенов текущего контекста. При генерации следующего токена повторное вычисление K и V для предшествующих токенов не требуется — они читаются из кэша.

**Вычислительная сложность:**
```
Без KV-кэша: O(n²) — перевычисление всего контекста на каждом шаге
С KV-кэшем:  O(n)  — только новый токен + чтение кэша
```

**Объём KV-кэша** зависит от размера контекстного окна, количества слоёв и размерности скрытого состояния:

```
Приближённая оценка для Q4_K_M моделей:
  7B,  num_ctx 4096:  ~0.5 GB
  32B, num_ctx 4096:  ~2.0 GB
  32B, num_ctx 8192:  ~4.5 GB
  32B, num_ctx 32768: ~18 GB
```

При нехватке VRAM KV-кэш вытесняется в RAM, что приводит к снижению скорости генерации пропорционально разнице в bandwidth между VRAM и RAM.

---

### 3.5 Контекстное окно

**Контекстное окно (context window)** — максимальное количество токенов, доступных модели одновременно при генерации ответа. В контекстное окно включаются все компоненты диалога:

```
context = системный промпт
        + история диалога
        + текущий запрос пользователя
        + генерируемый ответ
        + retrieved chunks (при RAG)
        + tool calls история (при агентном режиме)
```

При превышении лимита ранние токены вытесняются по принципу скользящего окна — модель перестаёт их учитывать.


Размер контекстного окна напрямую определяет объём KV-кэша и суммарные требования к VRAM.

## 4. Движок модели

### 4.1 Назначение и компонентный состав

GGUF-файл содержит только статические данные — веса модели. Для генерации текста требуется inference runtime, выполняющий следующие операции:

1. **Загрузка весов** из файла в память (VRAM или RAM)
2. **Токенизация** входного текста — разбивка на токены с использованием словаря из GGUF
3. **Forward pass** — последовательное применение всех слоёв нейронной сети к входным токенам
4. **Сэмплирование** следующего токена — выбор из распределения вероятностей с учётом параметров temperature, top-p, top-k
5. **Детокенизация** — преобразование выходных токенов обратно в текст
6. **Управление KV-кэшем** — аллокация и инвалидация буферов в VRAM
7. **HTTP API** — приём запросов, формирование ответов, потоковая передача (streaming)

**llama.cpp** — низкоуровневая C++ библиотека, реализующая все перечисленные операции с поддержкой CUDA, ROCm и Metal бэкендов. Является основой большинства современных inference runtime.

**OpenAI-совместимый API** — стандартный интерфейс взаимодействия, позволяющий использовать локальный inference через унифицированный протокол:

```
POST /v1/chat/completions
POST /v1/completions
GET  /v1/models
```
Самый простой. Команда ollama run llama4:scout — модель уже отвечает в чате. Под капотом llama.cpp с GGUF-квантами, работает на чём угодно, включая Apple Silicon через MLX. Хорош для входа в llm inference, прототипа автоматизации и небольших команд.
Минусы вылезают под нагрузкой. По [бенчмаркам Red Hat 2026](https://developers.redhat.com/articles/2025/08/08/ollama-vs-vllm-deep-dive-performance-benchmarking), при 8 параллельных пользователях Ollama выдаёт 41 ток/сек, vLLM на том же железе — 793 ток/сек. Разница в 19 раз. P99 latency у Ollama — 673 мс, у vLLM — 80 мс.

**vLLM** — production inference сервер, написанный на Python поверх PyTorch. Реализует PagedAttention — управление KV-кэшем по аналогии с виртуальной памятью OS, что позволяет обслуживать сотни параллельных запросов без фрагментации памяти. Тот же OpenAI-совместимый API, но архитектура заточена под нагрузку: continuous batching (новые запросы добавляются в батч не дожидаясь завершения текущих), tensor parallelism (модель автоматически распределяется по нескольким GPU). Требует CUDA и не работает с GGUF (vLLM поддерживает AWQ и GPTQ квантизацию — это не GGUF, но тоже 4-битные веса в safetensors формате [пример](https://huggingface.co/Qwen/Qwen2.5-Coder-32B-Instruct-AWQ)) — использует исходные веса в формате safetensors/fp16.
---

### 4.2 Конфигурация движка (Modelfile)

**Modelfile** — декларативный файл конфигурации модели в inference runtime. Определяет источник весов, системный промпт и параметры сэмплирования.

```
FROM <путь к GGUF файлу>

SYSTEM "<системный промпт>"

PARAMETER num_gpu        <количество слоёв на GPU>
PARAMETER num_ctx        <размер контекстного окна>
PARAMETER temperature    <температура сэмплирования>
PARAMETER top_p          <параметр nucleus sampling>
PARAMETER top_k          <ограничение числа кандидатов>
PARAMETER repeat_penalty <штраф за повторение токенов>
```

**Разбор параметров:**

`FROM` — путь к GGUF файлу. Определяет загружаемую модель.

`SYSTEM` — системный промпт. Скрытая инструкция, передаваемая модели перед каждым диалогом. Не отображается пользователю. Определяет роль, стиль и ограничения поведения модели. Токенизируется и занимает место в контекстном окне при каждом запросе. Для модели имеет вид:

```
<|im_start|>system
{системный промпт}
<|im_end|>
<|im_start|>user
{запрос пользователя}
```

`num_gpu` — количество слоёв модели, выгружаемых на GPU. Значение 99 означает «все слои». При нехватке VRAM оставшиеся слои обрабатываются CPU. Производительность деградирует пропорционально доле слоёв, обрабатываемых вне GPU.

`num_ctx` — размер контекстного окна в токенах. Определяет объём KV-кэша в VRAM. Для агентных задач и RAG рекомендуется значение 8192 и выше.

---

### 4.3 Сэмплирование и детерминизм

При генерации каждого токена модель возвращает **logits** — вектор ненормализованных логарифмических вероятностей по всему словарю. После применения softmax получается распределение вероятностей:

```
"int"    → 35%
"void"   → 28%
"struct" → 15%
"char"   → 12%
...
```

**Сэмплер** выбирает один токен из этого распределения с учётом параметров:

`temperature` — масштабирует logits перед softmax: `logits_scaled = logits / temperature`. При temperature=0 выбирается токен с максимальной вероятностью (argmax) — вывод детерминирован и воспроизводим.

```
temperature = 0.0 → детерминированный (генерация кода)
temperature = 0.7 → стандартный режим
temperature = 1.5 → высокая вариативность
```

`top_k` — сэмплирование только из k наиболее вероятных токенов. Остальные исключаются из рассмотрения.

`top_p` (nucleus sampling) — сэмплирование из минимального набора токенов, суммарная вероятность которых ≥ p. При top_p=0.9 рассматривается минимальный набор с суммарной вероятностью 90%.

`repeat_penalty` — штраф за появление токенов, уже присутствующих в контексте. Снижает вероятность зацикливания генерации.

**Недетерминизм** при temperature>0 является намеренным свойством: два идентичных запроса могут дать разные ответы. Для воспроизводимой генерации (код, структурированные данные) рекомендуется temperature=0.


## 5. RAG: Retrieval-Augmented Generation

**RAG (Retrieval-Augmented Generation)** — архитектурный паттерн расширения LLM внешней базой знаний, при котором релевантные фрагменты документов динамически извлекаются из векторного хранилища и включаются в контекст каждого запроса. Предложен в работе Lewis et al. (Meta AI, 2020).

Паттерн решает фундаментальное ограничение параметрических языковых моделей: знания модели зафиксированы в весах на момент обучения и не обновляются без переобучения (fine-tuning). Fine-tuning требует значительных вычислительных ресурсов и не гарантирует сохранение ранее приобретённых знаний (catastrophic forgetting). RAG устраняет эту проблему без изменения параметров модели.

**Ключевое свойство:** в контекст запроса включаются не все документы базы знаний, а только семантически релевантные фрагменты — как правило 3-5 чанков суммарным объёмом 1500-2500 токенов. Остальные документы остаются в векторном хранилище и не участвуют в текущем запросе.

```
Входной контекст LLM = системный промпт
                     + retrieved chunks (3-5 фрагментов)
                     + история диалога
                     + запрос пользователя
```

Система работает в двух режимах: **офлайн** (индексирование, выполняется один раз) и **онлайн** (retrieval + generation, выполняется при каждом запросе).

```
Фаза 1: ИНДЕКСИРОВАНИЕ    ← один раз, офлайн
Фаза 2: RETRIEVAL         ← при каждом запросе
Фаза 3: AUGMENTATION      ← при каждом запросе
Фаза 4: GENERATION        ← при каждом запросе
```

---

### 5.1 Фаза индексирования

Выполняется однократно при загрузке документов. Результат персистируется в векторной базе данных.

#### Chunking

Документ разбивается на фрагменты фиксированного или переменного размера. Прямая векторизация документа целиком невозможна по двум причинам: эмбеддинг-модель имеет ограничение на длину входной последовательности (обычно 512-8192 токенов), а документ семантически неоднороден — единый вектор усредняет смысл всех разделов и теряет топическую специфику.

**Параметры:**

`chunk_size` — максимальный размер фрагмента в токенах.

`chunk_overlap` — перекрытие между смежными фрагментами в токенах. Устраняет потерю информации на границах разбивки.

```
chunk_size=512, chunk_overlap=50:

Chunk 1: токены [1 ... 512]
Chunk 2: токены [462 ... 973]   ← перекрытие 50 токенов
Chunk 3: токены [923 ... 1434]
```

**Стратегии разбивки:**

*Fixed-size* — разбивка по числу токенов без учёта семантических границ.

*Recursive character splitting* — иерархическая разбивка: сначала по параграфам (`\n\n`), затем по предложениям, затем по словам. Сохраняет смысловые единицы. Применяется в LangChain по умолчанию.

*Document-aware* — разбивка с учётом структуры документа (заголовки, разделы). Оптимально для структурированной технической документации.

*Semantic chunking* — разбивка в точках резкого изменения семантической близости между соседними предложениями. Максимальное качество, наибольшие вычислительные затраты.

**Структура чанка:**

```python
chunk = {
    "id":       "chunk_a1b2c3d4",
    "text":     "текст фрагмента",
    "metadata": {
        "source":      "document.pdf",
        "page":        47,
        "section":     "3.2 Initialization",
        "chunk_index": 143
    }
}
```

Поле `metadata` не участвует в векторном поиске — используется для атрибуции источника в ответе.

**Влияние параметров на качество retrieval:**

```
chunk_size 200 токенов:
  (+) высокая точность retrieval
  (-) недостаточный контекст вокруг целевого фрагмента

chunk_size 800 токенов:
  (+) полный контекст вокруг целевого фрагмента
  (-) сниженная точность retrieval
  (-) повышенная нагрузка на контекст LLM
```

---

#### Embedding (векторизация)

Каждый chunk преобразуется в числовой вектор фиксированной размерности с помощью **эмбеддинг-модели** — отдельной нейронной сети, обученной отображать семантически близкие тексты в близкие точки многомерного пространства.

> **Эмбеддинг-модель и LLM — независимые компоненты.** Эмбеддинг-модель выполняет только функцию кодирования текста в вектор, работает на CPU, не генерирует текст.

**Семантическое пространство:**

```
"инициализация UART"     → [0.23, -0.87, 0.41, ...]  ←┐ близко
"настройка serial port"  → [0.21, -0.85, 0.39, ...]  ←┘
"рецепт борща"           → [-0.54, 0.23, -0.71, ...]    далеко
```

Размерность вектора: 768, 1024 или 3072 чисел типа float32 (4 байта каждое).

**Примеры эмбеддинг-моделей:**

```
nomic-embed-text       
mxbai-embed-large      
bge-m3                
GigaEmbeddings (Сбер)   
text-embedding-3-large  
```

Индексирование — разовая операция. После завершения база готова к использованию без повторного пересчёта.

---

#### Хранение в векторной базе данных

Векторы, тексты и метаданные сохраняются в специализированной базе данных, поддерживающей поиск ближайших соседей в многомерном пространстве — **ANN (Approximate Nearest Neighbors)**.

**Физическое содержимое хранилища:**

```
Для каждого чанка:
  id:       уникальный идентификатор
  vector:   [float32 × N dim]
  text:     исходный текст чанка
  metadata: {source, page, section, ...}

Служебные структуры:
  HNSW-граф для быстрого ANN-поиска
```

**Векторные базы данных:**

```
ChromaDB   встроена в open-webui, SQLite под капотом, без настройки
Qdrant     production-ready, REST API, Docker
Weaviate   гибридный поиск (векторный + BM25 keyword)
pgvector   расширение PostgreSQL
FAISS      библиотека Meta, in-memory, без persistence
```

---

### 5.2 Фаза retrieval

Выполняется при каждом запросе.

#### Векторизация запроса

Запрос пользователя преобразуется в вектор той же эмбеддинг-моделью, что использовалась при индексировании. Это обязательное условие: векторы запроса и документов должны принадлежать одному метрическому пространству.

```
"как настроить последовательный интерфейс?"
    ↓ bge-m3 
[0.31, -0.72, 0.55, 0.18, ...]
```

#### ANN-поиск: алгоритм HNSW

Задача — найти `top-k` векторов в базе, наиболее близких к вектору запроса.

**Наивный подход (Brute Force):**

```
Сложность: O(N × D)
N=100 000 чанков, D=1024 dim → ~100 млн операций умножения
Время на CPU: ~50-100 мс — приемлемо для небольших баз
При N=10 млн: ~5-10 сек — неприемлемо
```

**HNSW (Hierarchical Navigable Small World):**

ChromaDB и Qdrant применяют HNSW — алгоритм построения многослойного графа, основанный на принципе «тесного мира»: в графе связей любые два узла достижимы через малое число промежуточных переходов.

```
Слой 2 (разреженный — длинные переходы):
  A ──────────────────── F ──── J

Слой 1 (средний):
  A ──── C ──── E ──── F ──── H ── J

Слой 0 (плотный — точный поиск):
  A─B─C─D─E─F─G─H─I─J─K─L─M
```

Каждый узел — вектор из базы. Поиск начинается на верхнем разреженном слое (быстрые переходы к нужному региону пространства), затем последовательно спускается к нижнему плотному слою (точный локальный поиск среди ближайших соседей).

```
Сложность HNSW: O(log N)
```

Компромисс: HNSW гарантирует приближённых, а не точных ближайших соседей. Параметры `ef_construction` (качество построения графа) и `M` (число рёбер на узел) регулируют баланс точность/скорость.

**Метрика расстояния — Cosine Similarity:**

```
cosine_similarity(A, B) = (A · B) / (||A|| × ||B||)

A · B = Σ(Aᵢ × Bᵢ)     — скалярное произведение
||A|| = √(Σ Aᵢ²)        — евклидова норма вектора

Диапазон: [-1, +1]
  +1.0  — векторы коллинеарны (идентичный смысл)
   0.0  — векторы ортогональны (несвязанный смысл)
  -1.0  — векторы антиколлинеарны
```

Cosine similarity измеряет угол между векторами, игнорируя их длину — это корректная метрика для семантической близости: короткий и длинный тексты об одном предмете дают коллинеарные векторы разной длины, но одинакового направления. Евклидово расстояние в этом случае ошибочно фиксировало бы их как несхожие.

**Результат retrieval:**

```
Запрос: "как настроить последовательный интерфейс?"

top-5 по cosine similarity:
  chunk_a1b2: 0.94  "UART configuration guide..."
  chunk_c3d4: 0.91  "Serial interface registers..."
  chunk_e5f6: 0.87  "Baud rate settings..."
  chunk_g7h8: 0.79  "Interface initialization..."
  chunk_i9j0: 0.71  "Port configuration..."
```

#### Reranking (опционально)

Двухэтапный retrieval повышает точность за счёт применения cross-encoder модели:

```
Этап 1: HNSW → top-20 кандидатов              
Этап 2: reranker(запрос, чанк) → top-5        
```

Cross-encoder рассматривает пару (запрос, чанк) совместно и формирует более точную оценку релевантности, чем независимые эмбеддинги запроса и документа. Применяется в production-системах где точность retrieval критична.

---

### 5.3 Фаза augmentation

Найденные фрагменты включаются в контекст запроса:

```
[system prompt]
  инструкция + указание опираться на предоставленный контекст

[retrieved context]
  === Source: document.pdf, page 47 ===
  [текст chunk 1]

  === Source: guide.pdf, page 12 ===
  [текст chunk 2]

  === Source: manual.pdf, page 8 ===
  [текст chunk 3]

[user message]
  запрос пользователя
```

---

### 5.4 Фаза generation

LLM генерирует ответ на основе предоставленного контекста. Два режима:

**Extractive** — модель извлекает информацию непосредственно из чанков без синтеза. Минимум галлюцинаций, ограниченная гибкость.

**Generative** — модель синтезирует ответ на основе нескольких чанков, перефразирует и объединяет информацию из разных источников. Требует более мощной модели.

**Контроль галлюцинаций:**

*Grounding* — системный промпт содержит явную инструкцию: «если ответ отсутствует в предоставленном контексте — сообщи об этом явно».

*Citation* — модель обязана указывать источник каждого утверждения. Позволяет пользователю верифицировать ответ.

---

### 5.5 Полная схема одного RAG-запроса

```
Запрос пользователя
        │
        ▼ 
Эмбеддинг-модель → вектор запроса
        │
        ▼ 
HNSW-поиск в векторной БД → top-k чанков
        │
        ▼ 
Reranker → отфильтрованный top-k
        │
        ▼ 
Формирование контекста: system + chunks + query
        │
        ▼ 
LLM inference → ответ со ссылкой на источник
```

---

### 5.6 Изоляция данных

```
База знаний:      общая для всех пользователей
                  переиндексируется только при добавлении документов

История диалога:  изолирована по аккаунтам
                  не попадает в контекст других пользователей
```

---

### 5.7 Факторы качества RAG

**Retrieval quality** определяет качество системы в целом. Если retrieval вернул нерелевантные чанки — даже мощная LLM не сформирует корректный ответ.

```
Факторы retrieval:
  1. Стратегия chunking        → сохранение смысловых единиц
  2. Качество эмбеддинг-модели → соответствие домену документов
  3. chunk_size и top_k        → баланс точности и полноты контекста
  4. Качество исходных документов
```

**Generation quality:**

```
Факторы generation:
  1. Размер LLM     → синтез из нескольких источников требует 14B+
  2. Системный промпт → явное ограничение на использование контекста
  3. num_ctx        → должен вмещать весь prefill + генерируемый ответ
```

---

### 5.8 Метрики оценки качества — RAGAS

RAGAS (Shahul Es et al., 2023) — фреймворк автоматической оценки RAG-систем. Четыре ключевые метрики:

**Context Recall** — полнота retrieval. Доля релевантной информации из эталонного ответа, присутствующей в retrieved чанках.

**Context Precision** — точность retrieval. Доля retrieved чанков, действительно полезных для формирования ответа.

**Faithfulness** — фактологическая верность ответа. Доля утверждений в ответе, подтверждённых retrieved контекстом.

**Answer Relevancy** — релевантность ответа исходному вопросу.

## 6. Агенты

### 6.1 Определение

**AI-агент** — это система, в которой LLM выступает не генератором финального текста, а **планировщиком действий**. Модель итеративно принимает решения о вызове внешних инструментов, получает результаты их исполнения как наблюдения и использует их для следующего шага рассуждения — до достижения конечного состояния задачи.

Формально: **Агент = LLM + Инструменты + Цикл (ReAct) + Память**

| | Обычная LLM | Агент |
|---|---|---|
| Источник данных | Только входной контекст | Контекст + результаты инструментов |
| Количество LLM-вызовов | 1 запрос → 1 ответ | N итераций до завершения |
| Взаимодействие с системой | Отсутствует | Через tool calls |

---
### 6.2 Инструменты

**Инструмент** — исполняемая функция с формализованным интерфейсом, доступная агенту через механизм tool calling. Реализуется на любом языке — Python, Go, Bash, C — это не имеет значения, важен только интерфейс.

Структура инструмента состоит из четырёх обязательных элементов:

- `name` — уникальный идентификатор
- `description` — семантическое описание. **Критически важно**: LLM принимает решение о вызове инструмента именно на основе этого текста, а не на основе кода
- `parameters` — JSON Schema входных аргументов
- `implementation` — исполняемый код

**Пример описания инструмента в формате JSON Schema:**

```json
{
  "type": "function",
  "function": {
    "name": "read_file",
    "description": "Reads the content of a file from disk and returns it as string",
    "parameters": {
      "type": "object",
      "properties": {
        "path": {
          "type": "string",
          "description": "Absolute or relative path to the file"
        }
      },
      "required": ["path"]
    }
  }
}
```

**Ключевое ограничение**: модель не исполняет инструменты. Она генерирует **намерение** (tool_call объект). Реальное исполнение — ответственность фреймворка.

---
### 6.3 Цикл ReAct

**ReAct** (Reasoning + Acting, Yao et al., Google 2022) — архитектурный паттерн чередования рассуждения и действия. Каждая итерация: **Thought → Action → Observation**.

```
Задача от пользователя
        ↓
┌─────────────────────────────────────┐
│  Thought                            │
│  LLM анализирует контекст и         │
│  формулирует план следующего шага.  │
│  Не возвращается пользователю.      │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│  Action                             │
│  LLM генерирует tool_call —         │
│  структурированное намерение        │
│  вызвать инструмент с аргументами.  │
└──────────────────┬──────────────────┘
                   ↓
         Фреймворк исполняет функцию
                   ↓
┌─────────────────────────────────────┐
│  Observation                        │
│  Результат исполнения добавляется   │
│  в контекст с role: "tool".         │
└──────────────────┬──────────────────┘
                   │
         ┌─────────┘  ← следующая итерация
         ↓
  Есть tool_calls? ──→ Да → снова Thought
         ↓ Нет
  Финальный ответ пользователю
```

**Условие завершения**: цикл продолжается пока LLM возвращает `tool_calls`. Завершается когда возвращается `content` без `tool_calls`, либо достигнут `max_iterations`.

**Накопление контекста**: вся цепочка Thought → Action → Observation добавляется в `messages array`. Каждый LLM-вызов получает полную историю — контекст растёт с каждой итерацией. Это прямое следствие для VRAM: при `num_ctx 2048` агентная сессия из 3-4 итераций с чтением файлов переполняет контекст. Для агентной работы рекомендуется `num_ctx 8192+`.

---

### 6.4 Tool Calling: полный технический цикл

### Шаг 1 — описание инструментов в запросе

```json
POST /v1/chat/completions
{
  "model": "qwen-coder-3b",
  "messages": [...],
  "tools": [{
    "type": "function",
    "function": {
      "name": "read_file",
      "description": "Reads file content",
      "parameters": {
        "type": "object",
        "properties": {
          "path": {"type": "string"}
        },
        "required": ["path"]
      }
    }
  }]
}
```

### Шаг 2 — ответ модели: tool_call вместо текста

```json
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": null,
      "tool_calls": [{
        "id": "call_abc123",
        "type": "function",
        "function": {
          "name": "read_file",
          "arguments": "{\"path\": \"drivers/spi.c\"}"
        }
      }]
    }
  }]
}
```

`content: null` — модель не генерирует текст, она генерирует намерение. `id` — уникальный идентификатор вызова, обязателен для сопоставления результата с вызовом при parallel tool calling.

### Шаг 3 — фреймворк исполняет функцию

```python
msg = response["choices"][0]["message"]

if msg.get("tool_calls"):
    for tool_call in msg["tool_calls"]:
        name = tool_call["function"]["name"]
        args = json.loads(tool_call["function"]["arguments"])

        # dispatch — словарь {имя_инструмента: callable}
        result = dispatch[name](**args)  # реальное исполнение на машине

        messages.append({
            "role": "tool",
            "tool_call_id": tool_call["id"],  # связь с вызовом
            "content": result
        })
    # → следующий LLM-вызов с обновлённым messages
```

### Шаг 4 — состояние messages после одной итерации

```
role: system    → "You are an agent. Tools: [...]"
role: user      → "найди утечки памяти в drivers/spi.c"
role: assistant → content: null, tool_calls: [{read_file, "spi.c"}]
role: tool      → tool_call_id: "call_abc123", content: "<800 строк кода>"
role: assistant → следующий Thought...
```

Именно этот `messages array` передаётся в каждом следующем LLM-вызове целиком. Память агента — это не что-то встроенное в модель, это накопленная история сообщений.

---

### 6.5 Память агента

LLM stateless по природе — каждый вызов независим. Память реализуется на уровне фреймворка тремя механизмами:

**In-context memory** — основной механизм. Полный `messages array` передаётся в каждом LLM-вызове. Ограничена `num_ctx`. При переполнении старые сообщения вытесняются и агент «забывает» начало задачи.

**External memory (Vector DB)** — результаты инструментов индексируются в векторной базе (ChromaDB, Qdrant). Перед каждым LLM-вызовом выполняется семантический поиск релевантных фрагментов. Позволяет работать с кодовыми базами превышающими контекстное окно.

**Episodic memory** — после завершения сессии LLM генерирует структурированное резюме выполненных действий. Это резюме инжектируется в системный промпт новой сессии. Обеспечивает continuity между независимыми запусками.

> **Важно**: KV-кэш Ollama — это оптимизация вычислений внутри одного вызова, не механизм памяти. Кэш сбрасывается при завершении сессии.

## Практика: **Aider** — консольный агент

### Что такое Aider

**Aider** — консольный AI-агент для работы с кодовой базой. Open-source.

В отличие от обычного чата с LLM, aider:
- Читает файлы проекта и понимает структуру репозитория
- Редактирует файлы напрямую — не просто генерирует текст
- Показывает diff перед применением изменений
- Интегрируется с git — автоматически коммитит изменения
- Работает полностью локально через Ollama — интернет не нужен

## 7. Подбор железа: AI-сервер 

## 7.1 Про AMD

При выборе GPU для LLM inference необходимо учитывать не только аппаратные характеристики, но и зрелость программного стека (совокупность библиотек, драйверов и инструментов).

**Исторический контекст.** NVIDIA выпустила платформу CUDA в 2006 году. AMD выпустила аналог — ROCm — только в 2016 году. За эти 10 лет весь экосистемный стек глубокого обучения (PyTorch, TensorFlow, llama.cpp, vLLM, Ollama) был написан и оптимизирован под CUDA. ROCm поддерживается как вторичный бэкенд и исторически уступает по стабильности и охвату оптимизаций. 

Это не ознает, что AMD плох - просто мало кейсов LLM inference на AMD и могут возьникнуть подводные камни.

## 7.2 Фильтры при выборе Видеокарты

### 7.2.1 Объём видеопамяти (VRAM)
**Значение:** Чем больше тем лучше), но более менее интересные вещи можно делать только с моделью от 14B параметров.
Ставим минимум на 24GB.

---

### 7.2.2 Тип видеопамяти
Тип памяти определяет bandwidth (пропускную способность — скорость передачи данных между GPU и памятью):

```
HBM2e (High Bandwidth Memory 2e):  ~2000 GB/s 
HBM2:                              ~900 GB/s   
GDDR7:                             ~800–1800 GB/s 
GDDR6X:                            ~700–1000 GB/s 
GDDR6:                             ~300–500 GB/s
GDDR5X:                            ~200–350 GB/s
GDDR5:                             ~160–300 GB/s 
```
Bandwidth критичен потому что LLM inference — memory-bound задача (задача, ограниченная скоростью памяти, а не вычислениями): на каждый генерируемый токен GPU читает все веса модели из памяти. Зависимость прямая:

```
tok/s ≈ Bandwidth / Размер_модели_в_байтах
```

---

### 7.2.3 Разрядность шины памяти (Bus Width)

Шина памяти — физическое количество параллельных линий передачи данных. Чем шире шина, тем выше bandwidth при той же частоте памяти:

```
4096 bit 
2048 bit 
384 bit  
256 bit  
128 bit  
```

Карты с шиной 128 bit при объёме 32 GB (Tesla M10) — признак мультичиповой конфигурации (несколько отдельных GPU на одной плате с раздельной памятью). Для LLM inference такая архитектура неприемлема: модель не может использовать суммарный объём памяти как единое пространство.

---

### 7.2.4 Частота GPU 
Для LLM inference частота GPU не является определяющим параметром. Задача memory-bound: GPU простаивает, ожидая данных из памяти, а не наоборот. Более высокая частота ускоряет prefill (обработку входного контекста), но разница незначительна по сравнению с влиянием наличия Tensor Cores.

---

### 7.2.5 Количество CUDA-ядер
CUDA-ядра — базовые вычислительные блоки GPU, каждое из которых выполняет одну операцию с плавающей точкой за такт. Для memory-bound decoding количество CUDA-ядер почти не влияет на скорость. Влияет на:

- prefill при длинных контекстах (RAG-запросы)
- batch inference (обслуживание нескольких пользователей одновременно)

---

### 7.2.6 Архитектура GPU

Архитектура определяет поколение Tensor Cores (специализированных блоков для матричного умножения, ускоряющих операции нейронных сетей в 5–25 (смотрим по поколению) раз по сравнению с CUDA-ядрами):

```
Архитектура   Год   Tensor Cores  
─────────────────────────────────
Blackwell     2024  5-е пок.      
Ada Lovelace  2022  4-е пок.      
Hopper        2022  4-е пок.      
Ampere        2020  3-е пок.      
Turing        2018  2-е пок.      
Volta         2017  1-е пок.      
Pascal        2016  отсутствуют   
Maxwell       2014  отсутствуют   
```
