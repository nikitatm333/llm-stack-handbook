## 1. LLM

**Что такое LLM**

LLM (Large Language Model, большая языковая модель) — это нейронная сеть, обученная на огромных объёмах текстовых данных с целью моделирования условного распределения вероятностей следующего токена в последовательности. Проще говоря: модель научилась предсказывать, какое слово (или часть слова) должно идти следующим
  
**Токен** — минимальная единица обработки текста в LLM. Это не всегда целое слово: токеном может быть часть слова, знак препинания или пробел. Например, слово "tokenization" может быть разбито на токены ["token", "ization"]. Средняя длина токена в английском тексте — около 4 символов, в русском — чуть больше. Это важно понимать при оценке стоимости и скорости работы модели: скорость измеряется в **tok/s** (токенов в секунду).

## 2. Архитектура локального LLM стека

```
┌─────────────────────────────────────────┐
│           Пользователи и доступ         │
│   (браузер, VS Code плагин, API клиент) │
└──────────────────┬──────────────────────┘
                   │ HTTP / WebSocket
┌──────────────────▼──────────────────────┐
│                  UI                     │
│            (open-webui :3000)           │
└──────────────────┬──────────────────────┘
                   │ REST API (OpenAI-compatible)
┌──────────────────▼──────────────────────┐
│              API слой                   │
│           (Ollama :11434)               │
└──────────────────┬──────────────────────┘
                   │ Загрузка весов, inference
┌──────────────────▼──────────────────────┐
│           Движок модели                 │
│    (llama.cpp внутри Ollama + CUDA)     │

```

Каждый слой выполняет строго определённую функцию и может быть заменён независимо от других. Например, можно поменять UI с open-webui на любой другой OpenAI-совместимый клиент, не трогая движок.

## 3. Модель LLM

### 3.1 Параметры модели (веса)

**Нейронная сеть** — это математическая функция, состоящая из последовательных слоёв линейных преобразований и нелинейных функций активации. Каждый слой принимает на вход вектор чисел и производит другой вектор чисел.

**Линейное преобразование** — это умножение входного вектора на матрицу весов. Математически: `y = W · x + b`, где:
- `x` — входной вектор (например, числовое представление токена)
- `W` — матрица весов (числа, которые модель выучила в процессе обучения)
- `b` — вектор смещений (bias)
- `y` — выходной вектор

**Тензор** — многомерный массив чисел. Скаляр (одно число) — это тензор нулевого ранга. Вектор — тензор первого ранга. Матрица — тензор второго ранга. В нейронных сетях данные представлены тензорами: например, батч из 8 последовательностей по 2048 токенов, каждый из которых представлен вектором из 4096 чисел — это тензор размерности [8, 2048, 4096].

**Параметр (вес)** — отдельное числовое значение с плавающей точкой внутри матрицы `W` или вектора `b`. Именно эти числа модель подбирает в процессе обучения методом градиентного спуска. Совокупность всех параметров и есть "то, чему модель научилась".

**Количество параметров** — основная характеристика размера модели:
- 1.5B параметров = 1 500 000 000 чисел
- 7B параметров = 7 000 000 000 чисел
- 32B параметров = 32 000 000 000 чисел

При полной точности (FP32, 4 байта на параметр) модель 7B занимает **28 GB**. Это слишком много для потребительских видеокарт, поэтому применяется квантизация.

---
### 3.2 Квантизация

**Квантизация** — это процесс снижения числовой точности представления весов модели с целью уменьшения занимаемой памяти и ускорения вычислений.

**Форматы точности:**

| Формат | Байт на параметр | Биты | Диапазон |
|---|---|---|---|
| FP32 (float32) | 4 | 32 | ±3.4 × 10³⁸ |
| FP16 (float16) | 2 | 16 | ±65504 |
| BF16 (bfloat16) | 2 | 16 | ±3.4 × 10³⁸ (меньше точность мантиссы) |
| INT8 | 1 | 8 | -128 до 127 |
| Q4_K_M | ~0.5 | ~4.5 | (нелинейная) |
| Q2_K | ~0.31 | ~2.6 | (нелинейная) |

**Как работает квантизация на примере Q4:**
Вместо хранения каждого веса как 32-битного числа с плавающей точкой, веса группируются блоками (например, по 32 значения). Для каждого блока вычисляется масштабный коэффициент (scale) и минимальное значение. Затем каждый вес представляется как 4-битное целое число (0-15), а при вычислениях восстанавливается приближённое значение: `вес ≈ int4_value × scale + min`.

**На что влияет квантизация:**

1. **Объём VRAM** — линейно снижается пропорционально битности:
   - 7B FP16 = ~14 GB
   - 7B Q4_K_M = ~4.5 GB
   - 7B Q2_K = ~3.0 GB

2. **Качество ответов** — снижается при агрессивной квантизации:
   - Q8 / Q6 — практически идентично FP16, потери незначительны
   - Q4_K_M — хороший баланс, рекомендуется как основной формат
   - Q2_K — заметная деградация, галлюцинации чаще

3. **Скорость inference** — при Q4 на CPU может быть быстрее FP16 из-за меньшего объёма данных для чтения из памяти

**Суффиксы в названиях GGUF файлов:**
- `Q4_K_M` — 4-битная квантизация, метод K-quants, размер M (medium)
- `Q2_K` — 2-битная квантизация, метод K-quants
- `Q4_0` — 4-битная квантизация, базовый метод (старый, хуже Q4_K_M)
- `_K_M`, `_K_S`, `_K_L` — Medium/Small/Large варианты одного метода

---

### 3.3 Форматы хранения моделей

**GGUF (GPT-Generated Unified Format)** — бинарный формат хранения весов модели, разработанный командой llama.cpp. Это единственный файл, содержащий:
- Все веса модели (квантизованные или нет)
- Метаданные архитектуры (количество слоёв, размер скрытого состояния, тип attention)
- Словарь токенизатора
- Гиперпараметры (контекстное окно, тип нормализации)

### 3.4 KV-кэш

**Механизм Attention** — ключевой компонент архитектуры Transformer (на которой основаны все современные LLM). При генерации каждого нового токена модель должна "посмотреть" на все предыдущие токены в контексте, чтобы учесть их при предсказании следующего. Математически это реализуется через механизм **Self-Attention**: для каждого токена вычисляются три вектора — **Query (Q)**, **Key (K)** и **Value (V)**.

**KV-кэш (Key-Value Cache)** — это буфер в памяти (VRAM), хранящий уже вычисленные векторы K и V для всех токенов в текущем контексте. Вместо повторного вычисления K и V для каждого токена при генерации следующего, модель читает их из кэша.

**Почему это важно:**
- Без KV-кэша генерация каждого нового токена требовала бы перевычисления всего контекста — O(n²) операций
- С KV-кэшем — только вычисление для нового токена и чтение кэша — O(n) операций

**Объём KV-кэша** зависит от:
- Размера контекстного окна (`num_ctx`) — количества токенов в "памяти" модели
- Количества слоёв модели
- Размерности скрытого состояния

### 3.5 Контекстное окно

**Контекстное окно (context window)** — максимальное количество токенов, которое модель может "видеть" одновременно при генерации ответа. Это включает:
- Системный промпт
- Всю историю диалога
- Текущий запрос пользователя
- Генерируемый ответ

Если суммарная длина превышает контекстное окно, более ранние токены "вытесняются" и модель их "забывает".

## 4. Движок модели

### 4.1 Зачем нужен движок

GGUF-файл сам по себе — это просто набор чисел на диске. Для того чтобы модель генерировала текст, нужна программа, которая:

1. **Загружает веса** из файла в память (VRAM или RAM)
2. **Токенизирует** входной текст — разбивает его на токены с помощью словаря из GGUF
3. **Выполняет forward pass** — последовательно прогоняет токены через все слои нейронной сети
4. **Сэмплирует** следующий токен — выбирает из распределения вероятностей с учётом параметров (температура, top-p, top-k)
5. **Детокенизирует** результат — преобразует токены обратно в текст
6. **Управляет KV-кэшем** — оптимизирует память
7. **Предоставляет API** — принимает запросы и отдаёт ответы

**Ollama** — это движок на базе **llama.cpp** (низкоуровневая C++ библиотека для inference), обёрнутый в удобный сервис с:
- OpenAI-совместимым REST API
- Systemd-сервисом (автозапуск)
- Менеджером моделей (ollama pull, ollama list)
- Автоматическим определением GPU (CUDA / ROCm / Metal)

---

### 4.2 Параметры конфигурации движка (Modelfile)

Modelfile — декларативный файл конфигурации модели в Ollama. Пример Modelfile для qwen-coder-7b:

```
FROM /home/tnv/ai_infratest/models/Qwen2.5-Coder-7B-Instruct-Q2_K.gguf
SYSTEM "You are an expert embedded Linux developer specializing in C programming,
        Linux kernel drivers, RTOS, HiSilicon hi3518 and Rockchip SoCs.
        Always write clean efficient C code with error handling and comments."
PARAMETER num_gpu 33
PARAMETER num_ctx 2048
```

**Разбор параметров:**

`FROM` — путь к GGUF файлу. Говорит Ollama какую модель загружать.

`SYSTEM` — **системный промпт**. Это скрытая инструкция, которую модель получает перед каждым диалогом. Пользователь её не видит, но она задаёт "роль" и "характер" модели. В нашем случае мы говорим модели, что она — эксперт по embedded Linux, и задаём стиль ответов. Системный промпт просто "подклеивается" в начало каждого запроса внутри контекстного окна. Для модели это выглядит как: <|im_start|>system...<|im_end|><|im_start|>user....

`num_gpu` — количество слоёв модели, выгружаемых на GPU. Модель состоит из N слоёв (у 7B модели — 32 слоя). Значение `99` означает "все слои на GPU". Если VRAM не хватает — часть слоёв остаётся на CPU и обрабатывается в RAM. Это замедляет inference, но позволяет запустить модель большего размера.

`num_ctx` — размер контекстного окна в токенах (описано выше).

**Дополнительные параметры (в примерах используем, но важно знать):**

`temperature` — "температура" сэмплирования. Значение от 0 до 2.
- 0.0 — детерминированный вывод, всегда выбирается наиболее вероятный токен (подходит для генерации кода)
- 1.0 — стандартное поведение
- 2.0 — высокая случайность, творческие но непредсказуемые ответы

`top_p` — nucleus sampling. Модель выбирает следующий токен из минимального набора токенов, суммарная вероятность которых ≥ top_p. Значение 0.9 означает "рассматривать только те токены, которые в сумме дают 90% вероятности".

`top_k` — ограничение числа кандидатов при сэмплировании. top_k=40 означает "рассматривать только 40 наиболее вероятных токенов".

`repeat_penalty` — штраф за повторение токенов, предотвращает зацикливание.

---

### 4.3 Сэмплирование и детерминизм

При генерации каждого токена модель выдаёт не один ответ, а распределение вероятностей по всему словарю (десятки тысяч токенов). Например:

```
"int"    → 35%
"void"   → 28%
"struct" → 15%
"char"   → 12%
...
```

**Сэмплер** выбирает один токен из этого распределения с учётом параметров temperature, top_p, top_k. Именно поэтому LLM не детерминирована — два одинаковых запроса могут дать разные ответы. При temperature=0 всегда выбирается максимальный элемент (argmax), что делает вывод детерминированным.

## 5. RAG: Retrieval-Augmented Generation

### Зачем нужен RAG

LLM ограничена знаниями из обучающих данных: она не знает ничего после даты обучения и не знает внутренних данных организации. Дообучить модель на своих данных (fine-tuning) — это недели работы и дорогостоящие GPU-вычисления, и даже после этого модель может «забыть» старые знания.

RAG решает задачу доступа к актуальным и приватным данным без переобучения: **нужные знания динамически подставляются в контекст каждого запроса** из внешней базы знаний.

---

### Фундаментальное заблуждение

```
Заблуждение:
Вся документация → засасывается в контекст → модель отвечает

Реальность:
Вся документация → хранится в векторной БД (офлайн, один раз)
Запрос → семантический поиск → 3-5 релевантных фрагментов → в контекст
```

При 10 000 страницах документации в контекст попадёт ~1500-2000 токенов — три фрагмента по 400-500 токенов. Остальное остаётся в базе.

---

### Четыре фазы RAG

```
Фаза 1: ИНДЕКСИРОВАНИЕ    ← один раз, офлайн, минуты/часы
Фаза 2: RETRIEVAL         ← при каждом запросе, ~40-100 мс
Фаза 3: AUGMENTATION      ← при каждом запросе, мгновенно
Фаза 4: GENERATION        ← при каждом запросе, секунды
```

---

### Фаза 1 — Индексирование

Происходит один раз при загрузке документов. Результат сохраняется в векторной БД и используется для всех последующих запросов.

#### 1.1 — Chunking (разбивка на фрагменты)

Документ нельзя целиком превратить в один вектор по двум причинам:

1. Эмбеддинг-модель имеет ограничение на длину входа (обычно 512-8192 токенов)
2. Документ семантически неоднороден — разные разделы об разном. Один вектор не передаст это разнообразие

Поэтому документ разбивается на куски — **chunks**.

**Параметры chunking:**

`chunk_size` — максимальный размер куска в токенах. Обычно 400-800.

`chunk_overlap` — перекрытие между соседними кусками в токенах.

```
Без overlap (плохо):
  Chunk 1: токены 1-512
  Chunk 2: токены 513-1024
  Проблема: важное предложение разрезано на границе

С overlap=50 (хорошо):
  Chunk 1: токены 1-512
  Chunk 2: токены 462-1024
  Предложение целиком попадает в один из чанков
```

**Стратегии разбивки:**

*Fixed-size* — режем по числу токенов. Быстро, но может разрезать посередине важного абзаца.

*Recursive character splitting* — пробует разбить сначала по параграфам (`\n\n`), потом по предложениям (`.`), потом по словам. Старается сохранять смысловые единицы. Default в LangChain.

*Document-aware* — учитывает структуру документа. Заголовок раздела — граница чанка. Лучший вариант для структурированной технической документации.

*Semantic chunking* — анализирует семантическую близость соседних предложений, разбивает там где смысл резко меняется. Самый качественный, самый дорогой.

**Что хранится в каждом чанке:**

```python
chunk = {
    "id":       "chunk_a1b2c3d4",
    "text":     "текст фрагмента документа...",
    "metadata": {
        "source":      "document.pdf",
        "page":        47,
        "section":     "3.2 Initialization",
        "chunk_index": 143,
    }
}
```

Metadata не участвует в поиске — нужна чтобы показать пользователю источник ответа.

**Влияние параметров на качество:**

```
Маленький chunk (200 токенов):
  + Точный retrieval — нашли ровно нужное место
  - Может не хватить контекста вокруг нужной информации

Большой chunk (800 токенов):
  + Больше контекста вокруг нужного места
  - Retrieval менее точный — много лишнего в чанке
  - Больше токенов в контексте LLM

Рекомендация: 400-600 токенов, overlap 50-100
```

---

#### 1.2 — Embedding (векторизация)

Каждый chunk прогоняется через **эмбеддинг-модель** — отдельную нейросеть, обученную превращать текст в числовой вектор так, чтобы семантически похожие тексты давали близкие векторы.

> **Эмбеддинг-модель и LLM — это разные модели.** Эмбеддинг-модель маленькая (~300 MB), работает на CPU, только преобразует текст в вектор. LLM большая, нужна GPU, генерирует текст.

**Что такое вектор:**

Представь пространство с N измерениями (осями). Каждый текст — точка в этом пространстве. Эмбеддинг-модель обучена размещать похожие по смыслу тексты рядом, а непохожие — далеко.

```
"инициализация UART"        → [0.23, -0.87, 0.41, 0.12, 0.67, ...]
"настройка serial port"     → [0.21, -0.85, 0.39, 0.14, 0.65, ...]  близко!
"рецепт борща"              → [-0.54, 0.23, -0.71, 0.88, -0.12, ...]  далеко
```

Размерность вектора зависит от модели — обычно 768, 1024 или 3072 числа. Каждое число — float32 (4 байта).

**Популярные эмбеддинг-модели:**

```
nomic-embed-text     274 MB   768 dim    локально (Ollama)   быстрая
mxbai-embed-large    670 MB  1024 dim    локально (Ollama)   лучше качество
bge-m3              1.2 GB   1024 dim    локально (Ollama)   лучший для RU+EN
text-embedding-3-small  —    1536 dim    OpenAI API
text-embedding-3-large  —    3072 dim    OpenAI API          наилучшее качество
```

**Скорость индексирования:**

```
nomic-embed-text на CPU:  ~100 чанков/сек
bge-m3 на CPU:            ~40 чанков/сек

При 75 000 чанков:
  nomic: ~12 минут
  bge-m3: ~30 минут
```

Это разовая операция. После неё база готова.

---

#### 1.3 — Хранение в векторной БД

Векторы вместе с текстами и metadata сохраняются в специализированную базу данных.

**Чем векторная БД отличается от обычной:**

Обычная БД ищет по точному совпадению:
```sql
SELECT * FROM docs WHERE section = 'initialization'
```

Векторная БД ищет **ближайших соседей в многомерном пространстве** — то есть находит векторы наиболее похожие на вектор запроса. Это называется **ANN — Approximate Nearest Neighbors** поиск.

**Физическое содержимое векторной БД:**

```
Для каждого чанка хранится:
  id:       уникальный идентификатор
  vector:   [0.23, -0.87, 0.41, ...]  ← N чисел float32
  text:     исходный текст чанка
  metadata: {source, page, section, ...}

Плюс структура индекса для быстрого поиска (HNSW граф)
```

**Объём на диске:**

```
100 000 чанков × 768 dim × 4 байта = ~300 MB (только векторы)
+ тексты чанков ~150 MB
+ HNSW индекс  ~80 MB
Итого:         ~530 MB
```

**Популярные векторные БД:**

```
ChromaDB    встроена в open-webui, без настройки, SQLite под капотом
Qdrant      production-ready, REST API, Docker, рекомендуется для prod
Weaviate    гибридный поиск (векторный + keyword одновременно)
Pinecone    облачный managed сервис
pgvector    расширение PostgreSQL
FAISS       библиотека Meta, in-memory, очень быстрая, без persistence
```

---

### Фаза 2 — Retrieval (поиск)

Происходит при каждом запросе. Состоит из двух шагов.

#### 2.1 — Векторизация запроса

Вопрос пользователя прогоняется через **ту же эмбеддинг-модель** что использовалась при индексировании. Это критично — векторы запроса и документов должны быть в одном пространстве.

```
"как настроить последовательный интерфейс?"
           ↓ эмбеддинг-модель
[0.31, -0.72, 0.55, 0.18, 0.63, ...]
```

#### 2.2 — ANN поиск (Approximate Nearest Neighbors)

Задача: найти top-k векторов в базе, наиболее близких к вектору запроса.

**Наивный подход — полный перебор (Brute Force):**

```
Для каждого из N векторов в базе:
  посчитать расстояние до вектора запроса
  запомнить top-k минимальных

Сложность: O(N × D), где D — размерность вектора
При N=100 000, D=768: ~77 млн операций умножения
На CPU: ~50-100 мс — приемлемо для небольших баз
```

При миллионах векторов полный перебор слишком медленный. Поэтому используются алгоритмы приближённого поиска — **ANN**.

**Алгоритм HNSW (Hierarchical Navigable Small World):**

Это то что использует ChromaDB и Qdrant по умолчанию. Идея заимствована из теории «малого мира» — в социальных сетях любые два человека связаны через 6 рукопожатий.

При индексировании строится многослойный граф:

```
Слой 2 (разреженный — «дальние связи»):
  A ─────────────────────── F ─── J

Слой 1 (средний):
  A ──── C ──── E ──── F ──── H ── J

Слой 0 (плотный — «ближние связи»):
  A─B─C─D─E─F─G─H─I─J─K─L─M
```

Каждый узел — это вектор из базы. Узлы соединены рёбрами с ближайшими соседями. На верхних слоях — длинные «прыжки», на нижних — точный поиск.

**Алгоритм поиска:**

```
1. Входим на верхний разреженный слой
   → быстро прыгаем к приблизительно нужному региону
   → O(log N) шагов

2. Спускаемся на слой ниже
   → уточняем поиск в более плотном графе

3. На нижнем плотном слое (слой 0)
   → точный поиск среди ближайших соседей узла
   → возвращаем top-k
```

Это даёт сложность O(log N) вместо O(N) при полном переборе. На 1 миллионе векторов — в 50 000 раз быстрее.

Компромисс: HNSW находит **приближённых** ближайших соседей, не гарантированно точных. Параметры `ef_construction` и `M` регулируют баланс между точностью и скоростью.

**Метрика расстояния — Cosine Similarity:**

Расстояние между векторами измеряется через косинусное сходство:

```
cosine_similarity(A, B) = (A · B) / (||A|| × ||B||)

где:
  A · B  = скалярное произведение = Σ(Aᵢ × Bᵢ)
  ||A||  = длина вектора = √(Σ Aᵢ²)

Результат: от -1 до +1
  +1.0  = векторы идентичны
   0.0  = векторы перпендикулярны (несвязанный смысл)
  -1.0  = векторы противоположны
```

Почему cosine а не евклидово расстояние:

```
Текст A: "инициализация UART"              → короткий вектор, направление X
Текст B: "Подробное руководство по
          инициализации UART включая
          все параметры и примеры"          → длинный вектор, то же направление X

Cosine similarity ≈ 0.95   ✓ правильно (одинаковый смысл)
Евклидово расстояние: большое  ✗ неправильно (разные длины векторов)
```

Cosine измеряет **угол** между векторами — направление, а не длину. Это правильная метрика для семантической близости текстов.

**Результат retrieval:**

```
Запрос: "как настроить последовательный интерфейс?"

Найдено top-5:
  chunk_a1b2: similarity=0.94  "UART configuration guide..."
  chunk_c3d4: similarity=0.91  "Serial interface registers..."
  chunk_e5f6: similarity=0.87  "Baud rate settings..."
  chunk_g7h8: similarity=0.79  "Interface initialization..."
  chunk_i9j0: similarity=0.71  "Port configuration..."
```

**Опциональный шаг — Reranking:**

После первичного векторного поиска можно применить **reranker** — отдельную модель которая более точно оценивает релевантность. Это двухэтапный поиск:

```
Этап 1: HNSW поиск → top-20 кандидатов  (быстро, ~10 мс)
Этап 2: reranker оценивает каждый из 20 → выбирает top-5  (точнее, ~200 мс)
```

Reranker — это cross-encoder: он смотрит на пару (запрос, чанк) вместе и даёт более точную оценку релевантности чем эмбеддинг. Используется в production системах где важна точность.

---

### Фаза 3 — Augmentation (формирование контекста)

Найденные фрагменты подставляются в запрос к LLM по шаблону:

```
┌─────────────────────────────────────────────────┐
│ [system prompt]                                 │
│ "You are an expert assistant. Answer based      │
│  on the provided context."                      │
│                                                 │
│ [retrieved context]                             │
│ "=== Source: document.pdf, page 47 ===          │
│ [текст чанка 1]                                 │
│                                                 │
│ === Source: guide.pdf, page 12 ===              │
│ [текст чанка 2]                                 │
│                                                 │
│ === Source: manual.pdf, page 8 ===              │
│ [текст чанка 3]"                                │
│                                                 │
│ [user question]                                 │
│ "как настроить последовательный интерфейс?"     │
└─────────────────────────────────────────────────┘
```

**Подсчёт токенов:**

```
Системный промпт:       ~200 токенов
3 чанка по 500 токенов: ~1500 токенов
История диалога:        ~300 токенов
Вопрос пользователя:   ~50 токенов
─────────────────────────────────────
Итого prefill:          ~2050 токенов
```

Это и есть весь контекст который получает LLM. Никакой «всей документации» — только то что нашёл retrieval.

---

### Фаза 4 — Generation (генерация ответа)

LLM получает сформированный контекст и генерирует ответ, опираясь на найденные фрагменты.

```
Хороший RAG-ответ:
  "Согласно document.pdf (стр.47), для настройки
   последовательного интерфейса необходимо:
   1. Установить baud rate в регистре 0x4000...
   2. Включить модуль командой..."

Плохой ответ (без RAG или неудачный retrieval):
  "Обычно последовательный интерфейс настраивается
   через регистры управления..." ← галлюцинация
```

**Два режима использования найденного контекста:**

*Extractive* — модель буквально извлекает информацию из чанков. Точно, минимум галлюцинаций, но может быть негибким.

*Generative* — модель синтезирует ответ на основе чанков, перефразирует, объединяет из нескольких источников. Гибче, но требует более мощной модели.

---

### Полная схема одного RAG-запроса

```
Пользователь вводит вопрос
              │
              ▼
┌─────────────────────────┐
│    Эмбеддинг-модель     │  Вопрос → вектор
│    (CPU, ~30 мс)        │  [0.31, -0.72, 0.55, ...]
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│     Векторная БД        │  HNSW поиск ближайших соседей
│     HNSW граф           │  100 000 векторов → top-5 чанков
│     (~10 мс)            │  similarity: 0.94, 0.91, 0.87...
└───────────┬─────────────┘
            │
            ▼ (опционально)
┌─────────────────────────┐
│      Reranker           │  top-20 → точнее оценить → top-5
│      (~200 мс)          │
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────────────────────────────┐
│         Формирование контекста                  │
│  [system] + [чанк1 + чанк2 + чанк3] + [вопрос] │
│  Итого: ~2000 токенов prefill                   │
└───────────┬─────────────────────────────────────┘
            │
            ▼
┌─────────────────────────┐
│    LLM на GPU           │  Prefill: ~2-15 сек
│    (Ollama)             │  Генерация: ~10-30 сек
└───────────┬─────────────┘
            │
            ▼
Ответ со ссылкой на источник
```

---

### Общая vs персональная база знаний

```
Индексирование (один раз, общее):
  Документы → ChromaDB
  Одна база на всех пользователей
  Переиндексирование только при добавлении новых документов

При каждом запросе каждого пользователя:
  Векторизация вопроса:   ~30 мс  (CPU)
  Поиск в БД:             ~10 мс  (HNSW)
  Formирование контекста: мгновенно
  LLM генерация:          15-60 сек  (GPU) ← единственное узкое место

Изоляция:
  База знаний:       ОБЩАЯ  — все видят одни документы
  История диалога:   ЛИЧНАЯ — контекст разговора изолирован по аккаунтам
```

---

### Что влияет на качество RAG

#### Retrieval quality (нашли ли нужные чанки)

Это важнее чем качество LLM. Если retrieval вернул нерелевантные чанки — даже GPT-4 не даст хорошего ответа.

```
Факторы влияющие на retrieval:
  1. Качество chunking           → не разрезать смысловые единицы
  2. Качество эмбеддинг-модели   → понимание домена (технический текст)
  3. chunk_size и top_k          → баланс точности и полноты
  4. Качество документов         → мусор на входе = мусор на выходе
```

#### Generation quality (правильно ли использовали найденное)

```
Факторы:
  1. Размер LLM        → 3B хуже синтезирует из нескольких источников чем 32B
  2. Системный промпт  → явно указать "отвечай только на основе контекста"
  3. num_ctx           → должен вмещать весь контекст + ответ
```

#### Hallucination control

RAG не устраняет галлюцинации полностью, но сильно снижает. Две основных стратегии:

*Grounding* — явная инструкция в системном промпте: «если ответа нет в предоставленном контексте — скажи что не знаешь».

*Citation* — обязать модель указывать источник каждого утверждения. Пользователь может проверить.

---

### Метрики оценки качества RAG (RAGAS)

RAGAS — фреймворк для автоматической оценки RAG систем. Четыре ключевые метрики:

**Context Recall** — нашёл ли retrieval нужные фрагменты? Измеряется как доля релевантной информации из эталонного ответа которая присутствует в найденных чанках.

**Context Precision** — насколько найденные чанки релевантны? Доля чанков в retrieved контексте которые действительно полезны для ответа.

**Faithfulness** — насколько ответ LLM соответствует найденным фрагментам? Не выдумывает ли модель то чего нет в чанках.

**Answer Relevancy** — насколько ответ релевантен исходному вопросу?

# Агенты

## Определение

**AI-агент** — это система, в которой LLM выступает не генератором финального текста, а **планировщиком действий**. Модель итеративно принимает решения о вызове внешних инструментов, получает результаты их исполнения как наблюдения и использует их для следующего шага рассуждения — до достижения конечного состояния задачи.

Формально: **Агент = LLM + Инструменты + Цикл (ReAct) + Память**

| | Обычная LLM | Агент |
|---|---|---|
| Источник данных | Только входной контекст | Контекст + результаты инструментов |
| Количество LLM-вызовов | 1 запрос → 1 ответ | N итераций до завершения |
| Взаимодействие с системой | Отсутствует | Через tool calls |

---
## Инструменты

**Инструмент** — исполняемая функция с формализованным интерфейсом, доступная агенту через механизм tool calling. Реализуется на любом языке — Python, Go, Bash, C — это не имеет значения, важен только интерфейс.

Структура инструмента состоит из четырёх обязательных элементов:

- `name` — уникальный идентификатор
- `description` — семантическое описание. **Критически важно**: LLM принимает решение о вызове инструмента именно на основе этого текста, а не на основе кода
- `parameters` — JSON Schema входных аргументов
- `implementation` — исполняемый код

**Пример описания инструмента в формате JSON Schema:**

```json
{
  "type": "function",
  "function": {
    "name": "read_file",
    "description": "Reads the content of a file from disk and returns it as string",
    "parameters": {
      "type": "object",
      "properties": {
        "path": {
          "type": "string",
          "description": "Absolute or relative path to the file"
        }
      },
      "required": ["path"]
    }
  }
}
```

**Ключевое ограничение**: модель не исполняет инструменты. Она генерирует **намерение** (tool_call объект). Реальное исполнение — ответственность фреймворка.

## Цикл ReAct

**ReAct** (Reasoning + Acting, Yao et al., Google 2022) — архитектурный паттерн чередования рассуждения и действия. Каждая итерация: **Thought → Action → Observation**.

```
Задача от пользователя
        ↓
┌─────────────────────────────────────┐
│  Thought                            │
│  LLM анализирует контекст и         │
│  формулирует план следующего шага.  │
│  Не возвращается пользователю.      │
└──────────────────┬──────────────────┘
                   ↓
┌─────────────────────────────────────┐
│  Action                             │
│  LLM генерирует tool_call —         │
│  структурированное намерение        │
│  вызвать инструмент с аргументами.  │
└──────────────────┬──────────────────┘
                   ↓
         Фреймворк исполняет функцию
                   ↓
┌─────────────────────────────────────┐
│  Observation                        │
│  Результат исполнения добавляется   │
│  в контекст с role: "tool".         │
└──────────────────┬──────────────────┘
                   │
         ┌─────────┘  ← следующая итерация
         ↓
  Есть tool_calls? ──→ Да → снова Thought
         ↓ Нет
  Финальный ответ пользователю
```

**Условие завершения**: цикл продолжается пока LLM возвращает `tool_calls`. Завершается когда возвращается `content` без `tool_calls`, либо достигнут `max_iterations`.

**Накопление контекста**: вся цепочка Thought → Action → Observation добавляется в `messages array`. Каждый LLM-вызов получает полную историю — контекст растёт с каждой итерацией. Это прямое следствие для VRAM: при `num_ctx 2048` агентная сессия из 3-4 итераций с чтением файлов переполняет контекст. Для агентной работы рекомендуется `num_ctx 8192+`.

## Tool Calling: полный технический цикл

### Шаг 1 — описание инструментов в запросе

```json
POST /v1/chat/completions
{
  "model": "qwen-coder-3b",
  "messages": [...],
  "tools": [{
    "type": "function",
    "function": {
      "name": "read_file",
      "description": "Reads file content",
      "parameters": {
        "type": "object",
        "properties": {
          "path": {"type": "string"}
        },
        "required": ["path"]
      }
    }
  }]
}
```

### Шаг 2 — ответ модели: tool_call вместо текста

```json
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": null,
      "tool_calls": [{
        "id": "call_abc123",
        "type": "function",
        "function": {
          "name": "read_file",
          "arguments": "{\"path\": \"drivers/spi.c\"}"
        }
      }]
    }
  }]
}
```

`content: null` — модель не генерирует текст, она генерирует намерение. `id` — уникальный идентификатор вызова, обязателен для сопоставления результата с вызовом при parallel tool calling.

### Шаг 3 — фреймворк исполняет функцию

```python
msg = response["choices"][0]["message"]

if msg.get("tool_calls"):
    for tool_call in msg["tool_calls"]:
        name = tool_call["function"]["name"]
        args = json.loads(tool_call["function"]["arguments"])

        # dispatch — словарь {имя_инструмента: callable}
        result = dispatch[name](**args)  # реальное исполнение на машине

        messages.append({
            "role": "tool",
            "tool_call_id": tool_call["id"],  # связь с вызовом
            "content": result
        })
    # → следующий LLM-вызов с обновлённым messages
```

### Шаг 4 — состояние messages после одной итерации

```
role: system    → "You are an agent. Tools: [...]"
role: user      → "найди утечки памяти в drivers/spi.c"
role: assistant → content: null, tool_calls: [{read_file, "spi.c"}]
role: tool      → tool_call_id: "call_abc123", content: "<800 строк кода>"
role: assistant → следующий Thought...
```

Именно этот `messages array` передаётся в каждом следующем LLM-вызове целиком. Память агента — это не что-то встроенное в модель, это накопленная история сообщений.

## Память агента

LLM stateless по природе — каждый вызов независим. Память реализуется на уровне фреймворка тремя механизмами:

**In-context memory** — основной механизм. Полный `messages array` передаётся в каждом LLM-вызове. Ограничена `num_ctx`. При переполнении старые сообщения вытесняются и агент «забывает» начало задачи.

**External memory (Vector DB)** — результаты инструментов индексируются в векторной базе (ChromaDB, Qdrant). Перед каждым LLM-вызовом выполняется семантический поиск релевантных фрагментов. Позволяет работать с кодовыми базами превышающими контекстное окно.

**Episodic memory** — после завершения сессии LLM генерирует структурированное резюме выполненных действий. Это резюме инжектируется в системный промпт новой сессии. Обеспечивает continuity между независимыми запусками.

> **Важно**: KV-кэш Ollama — это оптимизация вычислений внутри одного вызова, не механизм памяти. Кэш сбрасывается при завершении сессии.

# Практика: **Aider** — консольный агент

## Что такое Aider

**Aider** — консольный AI-агент для работы с кодовой базой. Open-source.

В отличие от обычного чата с LLM, aider:
- Читает файлы проекта и понимает структуру репозитория
- Редактирует файлы напрямую — не просто генерирует текст
- Показывает diff перед применением изменений
- Интегрируется с git — автоматически коммитит изменения
- Работает полностью локально через Ollama — интернет не нужен

## Ключевые команды

### Работа с файлами

| Команда | Описание |
|---|---|
| `/add drivers/uart.c` | Добавить файл в контекст агента |
| `/add drivers/` | Добавить всю папку |

### Основная работа

| Команда | Описание |
|---|---|
| `/ask <вопрос>` | Задать вопрос **без изменения файлов** |
| `/code <задача>` | Попросить написать или изменить код |
| `/architect <вопрос>` | Обсудить архитектуру без правок |

> **Важно:** `/ask` только отвечает текстом. `/code` реально изменяет файлы — показывает diff и спрашивает подтверждение.
