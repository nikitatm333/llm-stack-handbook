## 1.1 Целесообразность использования LLM в целом

**Что такое LLM**

LLM (Large Language Model, большая языковая модель) — это нейронная сеть, обученная на огромных объёмах текстовых данных с целью моделирования условного распределения вероятностей следующего токена в последовательности. Проще говоря: модель научилась предсказывать, какое слово (или часть слова) должно идти следующим
  
**Токен** — минимальная единица обработки текста в LLM. Это не всегда целое слово: токеном может быть часть слова, знак препинания или пробел. Например, слово "tokenization" может быть разбито на токены ["token", "ization"]. Средняя длина токена в английском тексте — около 4 символов, в русском — чуть больше. Это важно понимать при оценке стоимости и скорости работы модели: скорость измеряется в **tok/s** (токенов в секунду).

## 2. Архитектура локального LLM стека

```
┌─────────────────────────────────────────┐
│           Пользователи и доступ         │
│   (браузер, VS Code плагин, API клиент) │
└──────────────────┬──────────────────────┘
                   │ HTTP / WebSocket
┌──────────────────▼──────────────────────┐
│                  UI                     │
│            (open-webui :3000)           │
└──────────────────┬──────────────────────┘
                   │ REST API (OpenAI-compatible)
┌──────────────────▼──────────────────────┐
│              API слой                   │
│           (Ollama :11434)               │
└──────────────────┬──────────────────────┘
                   │ Загрузка весов, inference
┌──────────────────▼──────────────────────┐
│           Движок модели                 │
│    (llama.cpp внутри Ollama + CUDA)     │
└──────────────────┬──────────────────────┘
                   │ VRAM / RAM
┌──────────────────▼──────────────────────┐
│            Модель LLM                   │
│  (GGUF файл: веса + метаданные)         │
└─────────────────────────────────────────┘
```

Каждый слой выполняет строго определённую функцию и может быть заменён независимо от других. Например, можно поменять UI с open-webui на любой другой OpenAI-совместимый клиент, не трогая движок.

## 3. Модель LLM

### 3.1 Параметры модели (веса)

**Нейронная сеть** — это математическая функция, состоящая из последовательных слоёв линейных преобразований и нелинейных функций активации. Каждый слой принимает на вход вектор чисел и производит другой вектор чисел.

**Линейное преобразование** — это умножение входного вектора на матрицу весов. Математически: `y = W · x + b`, где:
- `x` — входной вектор (например, числовое представление токена)
- `W` — матрица весов (числа, которые модель выучила в процессе обучения)
- `b` — вектор смещений (bias)
- `y` — выходной вектор

**Тензор** — многомерный массив чисел. Скаляр (одно число) — это тензор нулевого ранга. Вектор — тензор первого ранга. Матрица — тензор второго ранга. В нейронных сетях данные представлены тензорами: например, батч из 8 последовательностей по 2048 токенов, каждый из которых представлен вектором из 4096 чисел — это тензор размерности [8, 2048, 4096].

**Параметр (вес)** — отдельное числовое значение с плавающей точкой внутри матрицы `W` или вектора `b`. Именно эти числа модель подбирает в процессе обучения методом градиентного спуска. Совокупность всех параметров и есть "то, чему модель научилась".

**Количество параметров** — основная характеристика размера модели:
- 1.5B параметров = 1 500 000 000 чисел
- 7B параметров = 7 000 000 000 чисел
- 32B параметров = 32 000 000 000 чисел

При полной точности (FP32, 4 байта на параметр) модель 7B занимает **28 GB**. Это слишком много для потребительских видеокарт, поэтому применяется квантизация.

---
### 3.2 Квантизация

**Квантизация** — это процесс снижения числовой точности представления весов модели с целью уменьшения занимаемой памяти и ускорения вычислений.

**Форматы точности:**

| Формат | Байт на параметр | Биты | Диапазон |
|---|---|---|---|
| FP32 (float32) | 4 | 32 | ±3.4 × 10³⁸ |
| FP16 (float16) | 2 | 16 | ±65504 |
| BF16 (bfloat16) | 2 | 16 | ±3.4 × 10³⁸ (меньше точность мантиссы) |
| INT8 | 1 | 8 | -128 до 127 |
| Q4_K_M | ~0.5 | ~4.5 | (нелинейная) |
| Q2_K | ~0.31 | ~2.6 | (нелинейная) |

**Как работает квантизация на примере Q4:**
Вместо хранения каждого веса как 32-битного числа с плавающей точкой, веса группируются блоками (например, по 32 значения). Для каждого блока вычисляется масштабный коэффициент (scale) и минимальное значение. Затем каждый вес представляется как 4-битное целое число (0-15), а при вычислениях восстанавливается приближённое значение: `вес ≈ int4_value × scale + min`.

**На что влияет квантизация:**

1. **Объём VRAM** — линейно снижается пропорционально битности:
   - 7B FP16 = ~14 GB
   - 7B Q4_K_M = ~4.5 GB
   - 7B Q2_K = ~3.0 GB

2. **Качество ответов** — снижается при агрессивной квантизации:
   - Q8 / Q6 — практически идентично FP16, потери незначительны
   - Q4_K_M — хороший баланс, рекомендуется как основной формат
   - Q2_K — заметная деградация, галлюцинации чаще

3. **Скорость inference** — при Q4 на CPU может быть быстрее FP16 из-за меньшего объёма данных для чтения из памяти

**Суффиксы в названиях GGUF файлов:**
- `Q4_K_M` — 4-битная квантизация, метод K-quants, размер M (medium)
- `Q2_K` — 2-битная квантизация, метод K-quants
- `Q4_0` — 4-битная квантизация, базовый метод (старый, хуже Q4_K_M)
- `_K_M`, `_K_S`, `_K_L` — Medium/Small/Large варианты одного метода

---

### 3.3 Форматы хранения моделей

**GGUF (GPT-Generated Unified Format)** — бинарный формат хранения весов модели, разработанный командой llama.cpp. Это единственный файл, содержащий:
- Все веса модели (квантизованные или нет)
- Метаданные архитектуры (количество слоёв, размер скрытого состояния, тип attention)
- Словарь токенизатора
- Гиперпараметры (контекстное окно, тип нормализации)

### 3.4 KV-кэш

**Механизм Attention** — ключевой компонент архитектуры Transformer (на которой основаны все современные LLM). При генерации каждого нового токена модель должна "посмотреть" на все предыдущие токены в контексте, чтобы учесть их при предсказании следующего. Математически это реализуется через механизм **Self-Attention**: для каждого токена вычисляются три вектора — **Query (Q)**, **Key (K)** и **Value (V)**.

**KV-кэш (Key-Value Cache)** — это буфер в памяти (VRAM), хранящий уже вычисленные векторы K и V для всех токенов в текущем контексте. Вместо повторного вычисления K и V для каждого токена при генерации следующего, модель читает их из кэша.

**Почему это важно:**
- Без KV-кэша генерация каждого нового токена требовала бы перевычисления всего контекста — O(n²) операций
- С KV-кэшем — только вычисление для нового токена и чтение кэша — O(n) операций

**Объём KV-кэша** зависит от:
- Размера контекстного окна (`num_ctx`) — количества токенов в "памяти" модели
- Количества слоёв модели
- Размерности скрытого состояния

### 3.5 Контекстное окно

**Контекстное окно (context window)** — максимальное количество токенов, которое модель может "видеть" одновременно при генерации ответа. Это включает:
- Системный промпт
- Всю историю диалога
- Текущий запрос пользователя
- Генерируемый ответ

Если суммарная длина превышает контекстное окно, более ранние токены "вытесняются" и модель их "забывает".

## 4. Движок модели

### 4.1 Зачем нужен движок

GGUF-файл сам по себе — это просто набор чисел на диске. Для того чтобы модель генерировала текст, нужна программа, которая:

1. **Загружает веса** из файла в память (VRAM или RAM)
2. **Токенизирует** входной текст — разбивает его на токены с помощью словаря из GGUF
3. **Выполняет forward pass** — последовательно прогоняет токены через все слои нейронной сети
4. **Сэмплирует** следующий токен — выбирает из распределения вероятностей с учётом параметров (температура, top-p, top-k)
5. **Детокенизирует** результат — преобразует токены обратно в текст
6. **Управляет KV-кэшем** — оптимизирует память
7. **Предоставляет API** — принимает запросы и отдаёт ответы

**Ollama** — это движок на базе **llama.cpp** (низкоуровневая C++ библиотека для inference), обёрнутый в удобный сервис с:
- OpenAI-совместимым REST API
- Systemd-сервисом (автозапуск)
- Менеджером моделей (ollama pull, ollama list)
- Автоматическим определением GPU (CUDA / ROCm / Metal)

---

### 4.2 Параметры конфигурации движка (Modelfile)

Modelfile — декларативный файл конфигурации модели в Ollama. Пример Modelfile для qwen-coder-7b:

```
FROM /home/tnv/ai_infratest/models/Qwen2.5-Coder-7B-Instruct-Q2_K.gguf
SYSTEM "You are an expert embedded Linux developer specializing in C programming,
        Linux kernel drivers, RTOS, HiSilicon hi3518 and Rockchip SoCs.
        Always write clean efficient C code with error handling and comments."
PARAMETER num_gpu 33
PARAMETER num_ctx 2048
```

**Разбор параметров:**

`FROM` — путь к GGUF файлу. Говорит Ollama какую модель загружать.

`SYSTEM` — **системный промпт**. Это скрытая инструкция, которую модель получает перед каждым диалогом. Пользователь её не видит, но она задаёт "роль" и "характер" модели. В нашем случае мы говорим модели, что она — эксперт по embedded Linux, и задаём стиль ответов. Системный промпт просто "подклеивается" в начало каждого запроса внутри контекстного окна. Для модели это выглядит как: <|im_start|>system...<|im_end|><|im_start|>user....

`num_gpu` — количество слоёв модели, выгружаемых на GPU. Модель состоит из N слоёв (у 7B модели — 32 слоя). Значение `99` означает "все слои на GPU". Если VRAM не хватает — часть слоёв остаётся на CPU и обрабатывается в RAM. Это замедляет inference, но позволяет запустить модель большего размера.

`num_ctx` — размер контекстного окна в токенах (описано выше).

**Дополнительные параметры (в примерах используем, но важно знать):**

`temperature` — "температура" сэмплирования. Значение от 0 до 2.
- 0.0 — детерминированный вывод, всегда выбирается наиболее вероятный токен (подходит для генерации кода)
- 1.0 — стандартное поведение
- 2.0 — высокая случайность, творческие но непредсказуемые ответы

`top_p` — nucleus sampling. Модель выбирает следующий токен из минимального набора токенов, суммарная вероятность которых ≥ top_p. Значение 0.9 означает "рассматривать только те токены, которые в сумме дают 90% вероятности".

`top_k` — ограничение числа кандидатов при сэмплировании. top_k=40 означает "рассматривать только 40 наиболее вероятных токенов".

`repeat_penalty` — штраф за повторение токенов, предотвращает зацикливание.

---

### 4.3 Сэмплирование и детерминизм

При генерации каждого токена модель выдаёт не один ответ, а распределение вероятностей по всему словарю (десятки тысяч токенов). Например:

```
"int"    → 35%
"void"   → 28%
"struct" → 15%
"char"   → 12%
...
```

**Сэмплер** выбирает один токен из этого распределения с учётом параметров temperature, top_p, top_k. Именно поэтому LLM не детерминирована — два одинаковых запроса могут дать разные ответы. При temperature=0 всегда выбирается максимальный элемент (argmax), что делает вывод детерминированным.

## 5. RAG: Retrieval-Augmented Generation
### 5.1 Что такое RAG
**RAG** (Retrieval-Augmented Generation) — это не часть модели. Это архитектурный паттерн вокруг модели. Суть: перед тем как отправить вопрос в модель, система автоматически находит релевантные фрагменты из наших документов (без знаний) и добавляет их в контекст запроса. Дообучить модель на своих данных — это дорого и долго. RAG решает это проще.

### 5.2 Что можно получить:
- Снижение галлюцинаций: Поскольку модель теперь опирается на конкретный текст, вероятность того, что она что-то выдумает, значительно снижается.
- Цитирование: RAG позволяет модели указывать, из какого именно документа взят ответ, что критически важно для корпоративных баз знаний.
- Векторный поиск: Технически поиск фрагментов обычно происходит не просто по ключевым словам, а по «смыслу» с помощью векторных баз данных.

### 5.3 Принцип работы
Без RAG запрос выглядит так:

```
[системный промпт] + [вопрос пользователя]
```

С RAG:

```
[системный промпт] + [найденные фрагменты документов] + [вопрос пользователя]
```

**Подробнее в нескольких шагах:**
- 1. Индексирование (один раз)
Документы разбиваются на куски по 200–500 токенов. Каждый кусок прогоняется через **эмбеддинг-модель** — небольшую нейросеть, которая превращает текст в вектор чисел (числовое представление смысла). Векторы сохраняются в **векторную базу данных** (ChromaDB, Qdrant).

> **Вектор** — это числовое представление смысла текста. Похожие по смыслу тексты дают близкие векторы в многомерном пространстве.

- 2. Поиск (при каждом запросе)

Вопрос пользователя тоже превращается в вектор той же эмбеддинг-моделью. Система ищет в базе ближайшие по смыслу векторы — **семантический поиск**, не по ключевым словам а по смыслу.

- 3. Генерация

Найденные куски подставляются в контекст и вместе с вопросом уходят в LLM. Модель отвечает уже с доступом к конкретным данным.

### 5.4 На каком уровне в картине LLM стека сидит RAG

RAG находится на уровне приложения — между пользователем и API модели. Сама модель не знает о RAG ничего — она просто получает более длинный и информативный запрос.

```
Пользователь
    ↓
RAG слой (open-webui)
    ↓ (1) векторный поиск в базе знаний
Векторная БД → релевантные куски документов
    ↓ (2) формирование запроса
[вопрос + найденные куски]
    ↓ (3) отправка в модель
Ollama API → LLM → ответ
    ↓
Пользователь
```
**В многие UI сервисы RAG уже встроен как отдельный инструмент**

# Практика
## Ollama

### Шаг 1 — установить Ollama:
```
curl -fsSL https://ollama.com/install.sh | sh
```

### Шаг 2 — настроить Ollama чтобы слушал все интерфейсы (нужно для Docker):

```
sudo mkdir -p /etc/systemd/system/ollama.service.d/

sudo tee /etc/systemd/system/ollama.service.d/override.conf << 'EOF'
[Service]
Environment="OLLAMA_HOST=0.0.0.0:11434"
Environment="OLLAMA_KEEP_ALIVE=-1"
EOF

sudo systemctl daemon-reload
sudo systemctl restart ollama
```
**Проверка:**
```
ss -tlnp | grep 11434
```

### Шаг 3 — загрузить модели из готовых GGUF файлов:

```
# Создать папку для Modelfile конфигов
mkdir -p ~/ai_infratest/modelfiles

# По желанию редактируем системные промпты
cat > ~/ai_infratest/modelfiles/qwen-coder-3b << 'EOF'
FROM /home/tnv/ai_infratest/models/qwen2.5-coder-3b-instruct-q4_k_m.gguf

PARAMETER num_gpu 99
PARAMETER num_ctx 2048
EOF

cat > ~/ai_infratest/modelfiles/qwen-coder-7b << 'EOF'
FROM /home/tnv/ai_infratest/models/Qwen2.5-Coder-7B-Instruct-Q2_K.gguf
```

## open-webui

### Шаг 5 — создать docker-compose.yml и запустить open-webui:
```
mkdir -p ~/ai_infratest/docker

cat > ~/ai_infratest/docker/docker-compose.yml << 'EOF'
services:
  open-webui:
    image: ghcr.io/open-webui/open-webui:main
    container_name: open-webui
    ports:
      - "3000:8080"
    environment:
      - OLLAMA_BASE_URL=http://host.docker.internal:11434
    volumes:
      - open-webui:/app/backend/data
    extra_hosts:
      - "host.docker.internal:host-gateway"
    restart: always
volumes:
  open-webui:
EOF

cd ~/ai_infratest/docker
docker compose up -d
```

```
// http://192.168.0.79:3000
docker ps
```
